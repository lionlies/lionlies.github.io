延伸閱讀：CRUD

CRUD in Rails

  • CRUD(Create/Read/Update/Destroy)是網站最基礎的結構，Rails中建立CRUD結構的規範是 MVC 和 RESTful。MVC是Model, View, Controller, 這樣的架構可以把前後端和資料庫的工作切開，各做各的，比較好開發和管理。RESTful則是透過一種HTTP verb搭配一種helper對應到特定的action,

    1. GET + groups_path => index action
    2. GET + group_path(group) => show
    3. GET + edit_group_path(group) => edit
    4. GET + new_group_path => new
    5. POST + groups_path => create
    6. PUT + group_path(group) => update
    7. DELETE + group_path(group) => destroy

其他的組合則不會有任何動作，這樣不管瀏覽器端傳來什麼樣的request, routing都會直接幫我們導向對應的action去運作後續流程，並且因為只有create, update destroy這三個action才會動到server端的資料，也就一併解決了資安問題。

自己撰寫的 helper 包裝 html

  • 在 Rails101 “自己撰寫 helper 包裝 html”中，有段內容是

目前 helpers directory 內有 posts_helper.rb 以及 account/posts_helper.rb 兩個 helper.rb，經過測試，

  1. 在 posts_helper.rb 以及 account/posts_helper.rb 中寫入名字相同內容不同的兩個 helpers, posts_helper.rb 的優先。
  2. 承1，account/posts/index.html.erb 不會優先 render account/posts_helper.rb 內的 helper
  3. 不只 posts_helper.rb 內的 helper 是全域變數，account/posts_helper.rb 內的也是。(同名的話 posts_helper.rb 優先，不同名則不存在打架問題)。

既然同時有 posts_helper.rb 以及 account/posts_helper.rb，我會設想 account/posts_helper.rb 也有作用，想請問什麼情況下會在不同 helper.rb 寫入不同的 helper? 謝謝。（目前只想到方便管理而已）

小蟹：就是方便管理沒錯。如果是同名 method，那他們應該要做一樣的事情，而不是在兩個 helper 分別定義一次。

JL: 那請問什麼情況下會把 helper 特別寫在 account/posts_helper.rb？

小蟹：只有 account/posts/ 下面的 view erb 檔案會用到的時候。

JL: 也就是類似namespace, 單純做分類，方便管理的用途嗎？

小蟹：是。甚至你可以自己另外開新的 helper 檔案，以好管理、找得到、放在符合邏輯的位置為主。

在 new.html.erb 及 edit.html.erb 中改成 <%= render "form", group: @ group, post: @post %> 。想請問的是，這個 posts 底下名為 _form 的 partial, 實務上會把不是 form 的部份，也就是 <div class="col-md-4 col-md-offset-4"> 一併包進去嗎？
